## Correctness findings

### 1. `next-env.d.ts` committed with dev-only types path — breaks typecheck ⛑ FIXED

**Severity: high.** Both apps' `next-env.d.ts` were committed importing
`./.next/dev/types/routes.d.ts`, which only exists after `next dev`. On a clean
checkout or after `next build` (which writes `.next/types/`, not `.next/dev/types/`),
`tsc --noEmit` fails with TS2307. Git history shows this exact line flip-flopping
across five commits on June 3–4 ("build error solved", etc.) as the file was
regenerated by whichever mode ran last and re-committed.

**Root cause (verified against Next 16.2.7 source,
`next/dist/lib/typescript/writeAppTypeDeclarations.js`):** Next writes
`import "./<distDir>/types/routes.d.ts"` where `distDir` is `.next` for build and
`.next/dev` for dev. The file is *mode-specific by design* and cannot be stably
committed.

**Fix:**
- Added `next-env.d.ts` to `.gitignore` (matches current create-next-app templates)
- `git rm --cached` on both `apps/mobile/next-env.d.ts` and `apps/admin/next-env.d.ts`
  (files stay on disk; Next regenerates them on every `dev`/`build`)
- Normalized the on-disk copies to the build path

This permanently ends the flip-flop commits. Note: a truly fresh clone won't have the
file until the first `pnpm dev`/`pnpm build` — same as any current Next project.

### 2. Email promises QR attachments that may not exist ⛑ FIXED

**Severity: high.** `receiptEmail.ts` unconditionally said "Your ticket QR code is
attached as a PNG (show at the door)" based on ticket quantity, but
`buildTicketQrAttachments` silently returned `[]` on query error, missing `qr_data`,
or render failure — the buyer could be told to show a QR that was never attached.

**Fix:** `buildReceiptEmailHtml`/`buildReceiptEmailText` now take the **actual
attachment count** (computed after the QRs are rendered) and omit the QR sentence
entirely when zero attachments shipped. The count in the copy can no longer disagree
with the attachments on the email.

### 3. Attachment count could disagree with the receipt (status-filter divergence) ⛑ FIXED

**Severity: medium (plausible).** `buildTicketQrAttachments` re-queried `tickets`
with `.eq('status','valid')` while the email's ticket count came from
`loadReceiptData`, which queries with **no status filter**. On a receipt re-send
after a refund/transfer (`maybeSendOrderReceiptEmail` retries failed sends), the
email would claim N QR codes but ship fewer.

**Fix (structural):** tickets now flow through `OrderReceiptData` — `loadReceiptData`
exposes a `tickets: ReceiptTicket[]` field (valid tickets only, `code` + `qrData`),
and `buildTicketQrAttachments(tickets)` renders from that single source. There is now
exactly **one** tickets query per receipt send, so the PDF, the email copy, and the
attachments can never disagree. This also removed the duplicate `tickets` SELECT and
the second service-role client per send (finding 6).

### 4. QR build hoisted outside the send try/catch — receipt email could be lost ⛑ FIXED

**Severity: medium (plausible).** `await buildTicketQrAttachments(...)` ran before
the try/catch wrapping `resend.emails.send`, so any exception escaping it would kill
the receipt+PDF email entirely (swallowed upstream by `trySendReceipt` with only a
`console.error`) — removing the old invariant that the email is always attempted.

**Fix:** the QR pipeline is wrapped with `.catch(() => [])` at the call site in
`sendOrderReceiptEmail` and each per-ticket render is individually caught inside
`buildTicketQrAttachments`. QR failure now degrades to "no attachment" (and, per
finding 2, the copy stops promising one); it can never block the receipt.

### 5. Sequential rendering on the checkout/webhook critical path ⛑ FIXED

**Severity: medium.** The call chain (`/api/checkout/complete` and the Stripe
webhook → `fulfillPaymentIntent` → `trySendReceipt` → `sendOrderReceiptEmail`) is
awaited end-to-end, and the change ran the PDF render (up to 12 s timeout), then each
ticket QR render, all sequentially.

**Fix:**
- QR PNG renders now run in parallel (`Promise.all` over tickets, per-ticket
  error tolerance preserved)
- The PDF render and the QR batch run **concurrently** — total latency is now
  `max(PDF, QRs)` instead of `PDF + QR₁ + QR₂ + …`

### 6. Duplicate tickets query + second service-role client per send ⛑ FIXED

**Severity: medium.** `loadReceiptData` already fetched every ticket for the order
with `select('*')` (including `code`/`qr_data`) and discarded them;
`buildTicketQrAttachments` then created a second `createServiceRoleClient()` and
re-queried the same rows. Fixed structurally by finding 3 — the attachment builder
no longer touches the database at all.

## Cleanup / reuse / altitude findings

### 7. Sidebar width constants duplicated across apps ⛑ FIXED

`AdminLayout.tsx` re-declared `SIDEBAR_WIDTH = 240` / `SIDEBAR_WIDTH_TABLET = 76`
with a comment "Match member app HofAppShell" — but `HofAppShell` kept the identical
values as private constants, so the two could silently drift.

**Fix:** added `sidebarWidth = { full: 240, rail: 76 }` to
`@hof/design-tokens` (`breakpoints.ts`, exported from the package index and the
`tokens` aggregate). Both `HofAppShell` and `AdminLayout` now consume the token —
one source of truth, drift impossible.

### 8. QR render styling duplicated four times ⛑ FIXED (mobile app)

`QR_RENDER_OPTIONS` in `qr.ts` duplicated inline options in
`TicketScreen.tsx` (identical), `TicketDetailPanel.tsx` and `SellAtDoorModal.tsx`
(admin — already drifted: `#0A0A08` dark, widths 320/200).

**Fix:** extracted `apps/mobile/src/lib/qrRenderOptions.ts` as a client-safe module
(no `node:crypto` import, so client components can use it). `qr.ts` and
`TicketScreen.tsx` both consume it. The admin call sites live in a different package
and intentionally render at different sizes/colors for on-screen panels — left as-is
(see "Not auto-fixed").

### 9. `qrLine`/`ticketPhrase` ternaries duplicated verbatim ⛑ FIXED

The singular/plural copy was repeated character-for-character in
`buildReceiptEmailHtml` and `buildReceiptEmailText`. Extracted `ticketPhrase(qty)`
and `qrAttachmentLine(qrCount)` helpers next to the existing `ticketQty()` — the
HTML and plain-text bodies can no longer drift apart.

### 10. Redundant `Buffer.from(buffer)` copy ⛑ FIXED

`QRCode.toBuffer` already returns a Node `Buffer` (`@types/qrcode@1.5.x`);
`Buffer.from(buffer).toString('base64')` allocated and copied the entire PNG per
ticket for nothing. Now `buffer.toString('base64')`.

### 11. Mobile drawer bypassed the new `sidebarPadding` ⛑ FIXED

The docked sidebar used the new `sidebarPadding` (`'2px 12px 16px'` in full mode)
but the mobile drawer hardcoded `'2px 12px'` — dropping the 16 px bottom padding for
the identical full-mode content. The drawer now uses `sidebarPadding`.

### 12. Redundant explicit `src` props on `HofLogoMark` ⛑ FIXED

All three admin call sites passed `src="/assets/hof-emblem.png"`, which is exactly
`HofLogoMark`'s default. Removed — matching how the member app's `HofAppShell`
already calls it.

## Documented, not auto-fixed

### A. Login logo pixel-hack stack (asset has baked-in whitespace)

`login/page.tsx` wraps `HofLogoMark` in `overflow:hidden` + `lineHeight:0` +
`marginTop:-10` / `marginBottom:-6` + an inner `translateX(-8px)` div — **on top of**
the component's own internal `marginTop/-Bottom: -16`. `AdminLayout` renders the same
wordmark with no `translateX`, so two surfaces center the same logo differently.

**Why not fixed here:** the root cause is asymmetric whitespace baked into
`hof-emblem.png`. The right fix is re-cropping the asset (then deleting the
per-screen offsets and ideally the component's internal negative margins), which
needs a visual pass — blind CSS removal would shift the hand-tuned login layout.
**Recommendation:** re-crop the PNG tight to the artwork, then remove all
compensating offsets in one change and eyeball login + both sidebar modes.

### B. Brand emblem duplicated per app

`apps/admin/public/assets/hof-emblem.png` is byte-identical (SHA-256 match) to
`apps/mobile/public/assets/hof-emblem.png`. Next.js serves static assets from each
app's own `public/`, so a physical copy per app is the pragmatic norm — but a rebrand
must now touch every app. **Recommendation:** when `packages/brand` materializes
(reserved in the repo map), make it the canonical source and copy into each
`public/` at build time (e.g. a `prebuild` script), so the binaries can't drift.

### C. Admin on-screen QR styling differs from member/email QRs

`TicketDetailPanel.tsx` (`#0A0A08`, 320 px) and `SellAtDoorModal.tsx` (200 px) differ
from the shared `QR_RENDER_OPTIONS` (`#1a1a1a`, 400 px). Possibly intentional
(panel-sized display vs door-scannable print). If not intentional, point them at a
shared options export in `packages/ui` or `@hof/design-tokens`.

| `qr_data` could exceed QR capacity at EC level H and crash the render | `buildTicketQRData` produces a bounded ~150–180 char signed JSON payload (`code` + UUID + 64-hex HMAC); QR v40 at EC-H holds 1,273 alphanumeric chars. No write path produces oversized payloads. |
| Shared mutable `QR_RENDER_OPTIONS` object passed by reference | The `qrcode` lib does not mutate its options argument and nothing else does; no observable effect. |

Also checked and clean: `HofLogoMark` props all exist as used; `SendEmailAttachment`
shape matches Resend's base64-string contract; `tickets` columns exist in
`database.types.ts`; tickets are inserted before the receipt email sends
(`fulfillPayment.ts`); receipt routes default to the Node runtime (Buffer/`toBuffer`
safe); Resend's 40 MB limit is never approached (~2–5 KB per QR PNG).
